# Предложения изменений для ускорения `script.sql` (DB2 for z/OS)

Ниже несколько вариантов решений, от самых “лёгких” (без изменения запроса) до “надстроек” над таблицами. Скрипт сейчас тяжёлый из‑за двух вещей:
- фильтры по `SUBSTR(NrAccount,9,…)` (плохо индексируется)
- выборка из `PBI."InfoYSR"` по `DtBalance` при очень большом объёме

## Вариант 0 — без правок SQL (только обслуживание)
Подходит, если иногда план “плывёт” из‑за статистики.
- **RUNSTATS/статистика**: обновить статистику и распределения по ключевым колонкам:
  - `PBI."Account"(NrBank, NrAccount, NrEWallet, CdCurrency)`
  - `PBI."AccountStatus"(IDNAccount, AccountStatus, DtAccountOpen, DtAccountChange, StatusOwner)`
  - `PBI."InfoYSR"(DtBalance, NrAccount, NrEWallet, CdCurrency, NrBank, BalanceStatus)`
  - `PBI."SPAccountControl"(PrYSR, count_BalAccount, BalAccount)`
- **REORG** (если фрагментация) по самым горячим tablespace/indexspace.
- **Проверка актуальности пакетов/REBIND** (если запрос исполняется через пакет; иногда помогает при смене статистики).

## Вариант 1 — новые индексы (SQL не меняем)
Ты писал, что существующие индексы менять нельзя, но **новые создавать можно** — это самый “минимальный” и часто самый эффективный путь.

### 1.1. `PBI."InfoYSR"`: индекс с ведущим `DtBalance`
Цель: сделать фильтр `WHERE DtBalance = <одна дата>` максимально селективным и “ведущим”.

- **Предложение индекса (пример)**:
  - ведущие: `DtBalance` (DESC/ASC — не критично при равенстве), затем то, что участвует в формировании `Account`:
  - `DtBalance`, `NrAccount`, `NrEWallet`, `CdCurrency`
  - опционально добавить `NrBank` и/или `BalanceStatus`, если они помогают плану/фильтрам на вашем окружении.

Почему: текущий индекс начинается с `IDNInfoYSR`, поэтому Db2 часто не может “войти” по `DtBalance` и вынужден читать слишком много.

### 1.2. `PBI."SPBICBY"`: индекс под предикат
Таблица маленькая, но индекс всё равно может убрать лишние действия оптимизатора и стабилизировать план.
- **Индекс**: (`CdActRecord`, `BICStatus`, `NrBank`)

### 1.3. `PBI."SPAccountControl"`: индекс под подзапросы
Таблица маленькая (172 строки), но т.к. она используется 8 раз, индекс/статистика тоже может помочь.
- **Индекс**: (`PrYSR`, `count_BalAccount`, `BalAccount`)

### 1.4. `PBI."AccountStatus"`: индекс под фильтр по датам/статусам
В `AAC` есть условия по:
- `DtAccountOpen <= DtBalance`
- `DtAccountChange >= DtBalance` (для статусов 8/9)
- `AccountStatus IN (...)`
И всё это после соединения по `IDNAccount`.

Если план показывает “плохой” доступ к `AccountStatus`, пробовать:
- **Индекс**: (`IDNAccount`, `AccountStatus`, `DtAccountOpen`, `DtAccountChange`)

## Вариант 2 — вычисляемые (generated) колонки + индексы (SQL почти не меняем)
Цель: убрать `SUBSTR(NrAccount,9,…)` из предикатов так, чтобы Db2 мог эффективно использовать индекс.

### 2.1. Generated columns в `PBI."Account"` и `PBI."InfoYSR"`
Добавить вычисляемые колонки:
- `BalPrefix1 = SUBSTR(NrAccount, 9, 1)`
- `BalPrefix2 = SUBSTR(NrAccount, 9, 2)`
- `BalPrefix3 = SUBSTR(NrAccount, 9, 3)`
- `BalPrefix4 = SUBSTR(NrAccount, 9, 4)`

И затем индексы по этим колонкам (как минимум на той таблице, которая “самая больная” по плану):
- для `Account`: (`NrBank`, `BalPrefix4`) или (`BalPrefix4`, `NrBank`) — выбрать по плану
- для `InfoYSR`: (`DtBalance`, `BalPrefix4`) или (`BalPrefix4`, `DtBalance`) — выбрать по плану

Плюс: SQL можно оставить почти тем же, заменив `SUBSTR(NrAccount,9,4)` на `BalPrefix4` (минимальная правка).

## Вариант 3 — “таблица-помощник” со списком допустимых префиксов (SQL минимально)
Сейчас `SPAccountControl` дергается 8 раз и логика “1..4 символа” повторяется.

### 3.1. Подготовленная таблица `PBI."YSR_BalPrefixes"`
Создать таблицу со всеми валидными префиксами в одном столбце:
- `BalPrefix VARCHAR(4)`
- `BalPrefixLen SMALLINT`
- `PrYSR SMALLINT`

Заполнять из `SPAccountControl` (или поддерживать триггером/процедурой).

Тогда в SQL вместо 4 подзапросов можно сделать один JOIN по `BalPrefix` (это уже правка SQL, но точечная).

## Вариант 4 — View/таблица-обёртка с предрасчётами ключа `Account` (SQL почти не меняем)
В запросе ключ `"Account"` строится одинаково в `AA` и `SRA`.

### 4.1. View `PBI.V_AccountKey_Account`
View над `PBI."Account"`:
- добавляет вычисляемую колонку `AccountKey` (тот же CASE)
- добавляет префиксы `BalPrefix1..4`

Индексировать базовую таблицу/генерируемые колонки (см. вариант 2).

Дальше правка скрипта минимальная: заменить формирование `"Account"` на чтение готового `AccountKey` из view.

### 4.2. View `PBI.V_AccountKey_InfoYSR`
Аналогично для `InfoYSR`.

## Вариант 5 — Материализация (MQT/стейджинг) под ежедневный запуск
Если можно добавить подготовительный шаг (или делать это один раз в сутки заранее), это часто даёт самый большой выигрыш.

### 5.1. Таблица-слепок `PBI.YSR_EligibleAccounts_<date>`
Ежедневно собрать небольшой набор “актуальных” счетов:
- применить фильтры по банку
- применить фильтр по балансовым префиксам
- сразу собрать `AccountKey`
Индексировать эту таблицу по `AccountKey` и `IDNAccount`.

Тогда основной запрос будет работать только по “маленькой” таблице, а не по 155M в `Account`.

### 5.2. (Если доступно) MQT по `InfoYSR` на `DtBalance`
Суть: сделать “проекцию” `InfoYSR` по нужным полям и с хорошим ключом по `DtBalance` + `AccountKey`.
Это особенно полезно, если в `InfoYSR` много лишних колонок/широкие строки.

## Вариант 6 — минимальная правка SQL (часто ускоряет без DDL)
Если DDL менять тяжело/долго согласовывать, а SQL разрешено тронуть чуть-чуть:
- заменить `EXCEPT` на anti-join (`LEFT JOIN ... WHERE sra.Account IS NULL`)
- заменить `IN (SELECT ...)` на `JOIN`/`EXISTS` (Db2 часто лучше оптимизирует)

Этот вариант зависит от плана выполнения — без EXPLAIN можно легко сделать “хуже”.

## Что выбрать первым (порядок)
1) Получить **EXPLAIN** текущего запроса и понять, где самая большая стоимость (обычно `InfoYSR` или `Account`).
2) Если “болит `InfoYSR` по DtBalance” — начать с **нового индекса с ведущим `DtBalance`** (вариант 1.1).
3) Если “болит `SUBSTR`” — идти в **generated columns + индексы** (вариант 2) или “таблицу префиксов” (вариант 3).

## Что мне нужно, чтобы выдать точный DDL (без гаданий)
- текст EXPLAIN (или скрин visual explain с узлами: scan type, join method, estimated rows)
- фактическая `DtBalance` и оценки `COUNT(*)` для `AA/AC/AAC/SRA/AACS/Q1`
